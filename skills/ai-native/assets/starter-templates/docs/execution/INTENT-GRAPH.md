## 1. 意图状态机

```
            ┌──────────┐    收到任何关于这条意图的真实使用证据
   写入 ──> │ pending  │ ─────────────────────────────────────┐
            └──────────┘                                       │
                                                              ▼
            ┌──────────┐    样本量 < 阈值，但已有信号
            │ probing  │ <─────────────────────────────────────┤
            └──────────┘                                       │
                 │ 证据 ≥ 阈值                                   │
                 │ 或 user override                              │
                 │              证伪条件命中                     │
                 ▼                                              ▼
            ┌──────────┐                               ┌──────────┐
            │validated │ ──> 进入下一个 Batch ──>      │falsified │
            └──────────┘                               └──────────┘
                 │                                              │
                 ▼                                              ▼
            ┌──────────────┐                          反向修剪 docs/0X
            │ implemented  │                          + 立 ADR
            └──────────────┘
```

### 1.1 User Override（紧急通道）

用户明确表态"现在就要"时，可直接 `pending → validated`，但必须：
- 意图卡末尾添 `**User Override**：YYYY-MM-DD · 原因：<一句话>`
- 在 DECISIONS.md 立 ADR 记 override 决策
- 实施完成后保留 override 标记

**禁止**当默认通道。窗口内 override 比例 > 50% 视为范式失败。

### 1.2 三条 Fast Path

| 路径 | 适用场景 | 走 INTENT-GRAPH？ |
|------|----------|-------------------|
| A 探索 | "可能需要但不确定" | ✅ 必须 |
| B 紧急 | "我现在就想要" | ✅ 走 override 通道 |
| C 修缮（bug） | 修复已有功能错误 | ❌ 普通 PR |
| C 修缮（产品改动） | 改变已有功能语义 | ⚠️ 看是否触发 docs 修订 |

## 1.3 意图卡书写规范

### 1.3.1 可观测性硬规则（最重要）

**证据 / 证伪条件必须**全部可观测：

- 数据库查询能返回数字
- Prometheus / Grafana 指标存在或能补
- 应用日志 grep 可数
- 用户反馈但**必须量化**（"≥ 3 条独立消息"+原文记录）

**禁止主观词**：
- ❌ "用户体验糟糕" / "作者不满" / "界面不好用" / "需要时找不到"

**正确改写**：
- ❌ "用户找不到设置入口" → ✅ "应用日志 `event=ui.search query=settings` ≥ 5/周"
- ❌ "作者觉得卡顿" → ✅ "前端 RUM `LCP > 2.5s` 占比 > 20%"

### 1.3.2 数量上限

**Active 意图（pending + probing + validated）≤ 12 条**。

超出阈值时：合并 → falsify → 才考虑扩张上限（需立 ADR）。

`implemented` / `falsified` 不计入。

### 1.3.3 强制反问

用户用功能式语言（"我要 X 页面"）描述时，Agent 起草前**必须先反问 ≥ 1 轮**：
- 你为什么需要这个？
- 什么场景下会用？什么场景下不会用？
- 如果不做，用户怎么解决同一问题？

### 1.3.4 字段强制清单

```
### Intent #NNN · <一句话标题，意图导向>
**假设**：<一段话，回答"我们相信什么">
**对应规格差异**：<docs/0X 的具体引用，或"无对应规格——新意图">
**当前替代**：<不实施时用户怎么完成同一目标>
**证据触发条件**（任一即视为有信号）：<可观测条目>
**证伪条件**：<可观测条目>
**实施空间**（不锁死）：<极简 / 中量 / 重量三档>
**默认状态**：`pending` | `probing`
**Probing 信号**：<空，或时间戳 + 信号原文>
**User Override**：<空，或 YYYY-MM-DD · 原因>
```

## 4. 更新规则

| 谁 | 何时 | 做什么 |
|----|------|--------|
| 用户 | 任意时刻 | 改任意意图状态 / 加新意图 / 删意图 / 行使 override |
| 用户 | user override 时 | 添 `User Override` 字段 + DECISIONS.md 立 ADR |
| Agent | 每个 Batch 完成时 | 回写 §3 对账表，已实施意图状态置 `implemented` |
| Agent | 收到真实信号时 | 升 `probing`，**Probing 信号**加时间戳 + 原文 |
| Agent | 检测到证伪条件命中 | 状态置 `falsified`，反向修剪 docs/0X，立 ADR |
| Agent | 收到功能式需求时 | 强制反问 ≥ 1 轮（§1.3.3） |
| Agent | active 意图 > 12 时 | 优先合并 → falsify → 才申请扩张 |

**禁止**：
- 把 INTENT-GRAPH 当 PRD（无人天 / deadline / 甘特图）
- 不带证据 + 没有 override 就升 `validated`
- 在没证伪命中且没 override 时跳过本图谱去做规格里其他规划页面
- 主观词写证伪段
- override 比例 > 50%

## 5. 试点出口判定

| 条件 | 视为 |
|------|------|
| ≥ 1 条意图通过证据（非 override）走完 `pending → validated → implemented` | ✅ 范式成立 |
| ≥ 1 条意图明确 `falsified` 并按 ADR 反向修剪 docs/0X | ✅ 范式成立 |
| 窗口内 user override 比例 > 50% | ❌ 范式失败 |
| Active 意图无法收敛（持续 > 12 且合并 / falsify 都没发生） | ❌ 范式失败 |
| 6 周内无任何状态流转 | ❌ 范式失败 |

任一 ❌ 命中 → 立失败 ADR，停用本文件，回纯 Batch 范式。
所有 ✅ 命中 → 扩张到其他 docs 章节。
