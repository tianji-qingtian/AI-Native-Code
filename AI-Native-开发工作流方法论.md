# AI-Native 开发工作流方法论

## 0. 谁该读这份文档

- 用 Claude Code（或同类 AI agent CLI）作为主力开发工具的工程师
- 想把"AI-native 开发"落到实操而不是停留在概念的 tech lead
- 准备从零起新项目，或者把 AI agent 引入已有代码库的团队

如果你只是想用 Claude 改改代码，不要读——这套方法论的成本只有在中等规模以上项目才回本。

***

## 1. 核心诊断：那篇启发文章对在哪、错在哪

[`AI时代的编程范式革命：从人类逻辑PRD到AI-native自主规划.md`](AI时代的编程范式革命：从人类逻辑PRD到AI-native自主规划.md) 这篇文章主张：用「意图驱动 + Living Spec + 多 Agent 自主规划」替代传统 PRD。

**对的部分**：

- ✅ 静态 PRD 与代码必然脱节
- ✅ 单 Agent 上下文有限，多 Agent 协作是必然
- ✅ 机器可读契约（pytest / mypy / linter）应作真相源，不是 Markdown
- ✅ 规格应该 Living——能与代码双向同步

**错的部分**（照搬会撞墙）：

- ❌ "AI 可以自主推导实现路径"——当前 LLM 在长链路推理、跨上下文取舍、识别"自己不知道什么"上仍有显著失败率
- ❌ "意图清晰度是稀缺资源，实现能力不稀缺"——实现能力依然稀缺，只是分布不同；跨多服务的复杂集成 AI 仍会写出"能编译能过测试但语义错误"的代码
- ❌ "PRD = 人天估算 = 必须废除"——人天估算是 PRD 的副作用，不是本质。PRD 的本质（"可被审阅的产品意图载体"）永远需要存在，只是形式可以变
- ❌ 把"AI 自主"放在头部假设——会导致放任，最后产出的代码看起来都过测试，但产品线索糊掉

**结论**：这套范式真正的价值不是"让 AI 自主"，而是**强迫规格双向同步 + 让多 Agent 在共享契约下并行**。"自主"是修辞，"半自主"才是事实。

***

## 2. 责任分工：永远不要混淆

在任何阶段、任何场景下，下面这张分工表都成立。**搞错分工是这套方法论失败的最大原因**。

| 工作                     | 谁做                  | AI 能不能替代                          |
| ---------------------- | ------------------- | --------------------------------- |
| 商业判断 / 用户访谈 / 同理心      | 人类                  | ❌ 永远不能                            |
| 顶层意图定义                 | 人类                  | ❌ AI 可起草，人类必须定稿                   |
| 架构选型 / 技术栈 / 不可逆决策     | 人类                  | ❌ AI 可列选项，人类拍板                    |
| 收集"使用证据"（埋点 / 日志 / 反馈） | 人类（或埋点系统）           | ❌ AI 直接无法访问                       |
| 验收（"AI 报告全绿 ≠ 真的对"）    | 人类                  | ❌                                 |
| Batch 任务书起草            | AI                  | ✅ 可起草，人类审                         |
| 单 Batch 内的代码实施         | AI（Agent Team）      | ✅ AI 主力                           |
| 测试 / 类型 / linter 守门    | 工具自动化               | ✅ AI 配置 + 自动跑                     |
| 跨 Batch 全局一致性          | 机器可读契约              | ⚠️ 不靠 AI 记忆，靠 importlinter / mypy |
| 决策溯源                   | DECISIONS.md        | ✅ AI 起草，人类审                       |
| 规格双向同步                 | INTENT-GRAPH §3 对账表 | ✅ AI 回写，人类抽查                      |

**关键洞察**：把任何一行"人类做"的事委托给 AI，就是这套方法论失败的开始。

***

## 3. 通用 4-Phase 框架

无论空项目还是已有源码，工作流都遵循这 4 个阶段：

```
Phase 0  上下文凝练       人类主笔 + AI 反问
   │     空项目：顶层意图（前瞻）
   │     已有项目：codebase 考古（回顾）
   ▼
Phase 1  契约骨架         传统规格 + ADR
   │     空项目：架构 / 分层 / DB schema 第一版
   │     已有项目：反推意图 + 隐性约束显性化
   ▼
Phase 2  机器可读契约网络   测试 / 类型 / linter / importlinter
   │     这是质量保障的真相源，所有后续工作的安全网
   ▼
Phase 3  意图驱动 + 多 Agent  INTENT-GRAPH + Agent Team
         此时 AI 才能真正发挥规模优势
```

**两个铁律**：

1. **Phase 0/1 不能纯意图驱动**——AI 没有产品 / 没有代码可读，强行用意图卡会写出空想
2. **Phase 2 不能跳过**——没有契约网络，Phase 3 的 Agent Team 并行会破坏一切

## 4. 场景 A：空项目

### 4.1 起点

你脑里有个 idea，可能伴随：一段口语描述、一份 ChatGPT 生成的 PRD、几张原型图。

### 4.2 反模式（一开始就别这么干）

| 反模式                                       | 后果                          |
| ----------------------------------------- | --------------------------- |
| "Claude，根据这段描述生成 17 份规格"                  | 得到一份充满想象、与真实意图脱节的虚假 PRD     |
| "Claude，开始写代码吧"（跳过 Phase 0）               | AI 写出能跑但完全不是你要的东西           |
| Phase 0 就启用 INTENT-GRAPH                  | 没有产品没有用户，证据触发条件全是空想         |
| Phase 1 就开 9 个 Agent 并行                   | 没有契约网络，多 Agent 命名 / 抽象一致性必崩 |
| 项目纪律写进 \~/.claude/projects/<hash>/memory/ | 仓库一拷走纪律全丢                   |

### 4.3 Phase 0 · 意图凝练（1-3 天）

**输入**：自然语言愿景

**输出**：`docs/01-vision.md`（≤ 5 条顶层意图，每条带证伪条件）

**Claude Code 角色**：反问者 + 起草者

**关键差别**：顶层意图**不是功能清单**，是**可证伪的成功假设**。

| ❌ 顶层功能             | ✅ 顶层意图                           |
| ------------------ | -------------------------------- |
| "做一个 AI 写小说的 SaaS" | "作者能从 0 写完一本 ≥ 10 万字的完整小说，且质量稳定" |
| "支持多个 LLM 厂商"      | "作者切换 LLM 不需重新训练使用习惯，且成本可观测"     |
| "做用户管理后台"          | "租户 Owner 能在 5 分钟内邀请新成员并赋权"      |

**Claude Code 工作流**：

1. 用户用自然语言描述愿景（5-15 句话）
2. Claude 走强制反问：「为什么需要？什么场景下用？什么场景下不用？如果不做用户怎么解决？」
3. 反问 ≥ 1 轮后，Claude 起草 ≤ 5 条顶层意图
4. 用户审 / 改 / 砍 / 合并，定稿

**完成判定**：≤ 5 条顶层意图 + 每条都能用日志 / 指标 / 数据库查询验证

### 4.4 Phase 1 · 契约骨架（3-7 天，**必须传统范式**）

**为什么不能纯意图驱动**：

- AI 没有任何代码可读，无法做"扫描发现模式"
- 架构选型是不可逆的全局决策，等证据等不起
- 后续 Batch 需要稳定地基

**输出**：

- `CLAUDE.md`（项目纪律权威源；含硬约束、协作模式、技术栈、规划范式）
- `EXECUTION.md`（续跑剧本，下次新会话进来读什么）
- `docs/02-architecture.md`（一份精简架构图，**不是 17 份**）
- `docs/03-data-model.md`（数据库第一版 schema）
- `docs/execution/DECISIONS.md`（初始 5-10 条 ADR：技术栈选型 / 架构边界 / 不可逆决策）
- `docs/execution/AGENT-PROMPTS.md`（子代理模板，含通用公约）

**Claude Code 角色**：起草者 + 反问者

- "为什么用 PostgreSQL 而不是 Mongo？写到 DECISIONS.md 里"
- "DDD 分层吗？如果是，importlinter 契约是什么？"
- "前端 SPA 还是 SSR？现在不定后面会改不动"

**反模式**：让 AI 自己生成完整规格——它会编。Phase 1 必须是**人类主笔、AI 起草框架、人类填理由**。

**完成判定**：

- [ ] CLAUDE.md 已含项目硬约束（每条都能在某条 ADR 找到理由）
- [ ] 架构图能 1 屏装下，不超过 5 个核心组件
- [ ] DECISIONS.md ≥ 5 条 ADR
- [ ] 任何一个新会话读 CLAUDE.md + EXECUTION.md 能 5 分钟入门

### 4.5 Phase 2 · MVP 路线图 + 契约网络（1-4 周）

**输出**：

- `docs/execution/PROGRESS.md`（Batch 看板，依赖图）
- `docs/execution/BATCH-XX-*.md`（每个 Batch 一份）
- 测试 / 类型 / linter 配置（与代码同步建）
- MVP 主链路代码

**Batch 拆分原则**：

- ≤ 10 个 Batch（NovelForge 18 个偏多，因为有 17 份预制规格）
- 每个 Batch 验收的是"哪条顶层意图变得可 e2e 验证"，**不是**"规格 §3.4 功能列表"
- 严格依赖顺序——下游 Batch 依赖上游产出

**Claude Code 工作流**（每 Batch）：

1. 用户写 `BATCH-XX-*.md` / Claude 起草后用户审
2. 派 Agent Team
   - `general-purpose` 写代码
   - `Explore` 查约束 / 已有命名
   - `Plan` 设计实施路径
3. 完成后回写 `PROGRESS.md` + 必要时追加 `DECISIONS.md`
4. **每个 Batch 末尾人类必须验收**——AI 报告"全绿"≠ 真的对

**契约网络与代码同步建**：

- 每个 Batch 都要补对应测试（domain 90%、application 75%、infrastructure 60% 是 NovelForge 的取值，可调）
- mypy / linter / importlinter 在 Phase 1 就配，每 Batch 都跑
- pre-commit hook 强制执行

**INTENT-GRAPH 此时仍未启用**——它是 Phase 3 工具。

**完成判定**：

- [ ] 顶层意图至少 1 条 e2e 可验证
- [ ] CI 全绿（lint / type / test）
- [ ] 测试覆盖率达标
- [ ] DECISIONS.md 已记录所有 Batch 内的非平凡决策

### 4.6 Phase 3 · 意图驱动（永远不结束）

**触发条件**：MVP 主链路真跑通，开始有人用。

**此时 INTENT-GRAPH 才发力**：

- 用户使用过程中冒出"要不要补 X 页面"的零散需求 → 写成意图卡
- 加埋点 / 日志 / 指标，让证据通道真正能跑
- pending → probing → validated → implemented

**Agent Team 的真正用法**：

- 单意图卡 validated → 派 Agent Team 实施
- 同 Batch 多 agent 并行（NovelForge B1 9 个 agent 是上限，一般 3-5 个最佳）
- 必须读 AGENT-PROMPTS.md 通用公约的"多 Agent 并行命名一致性"段

**关键纪律**：

- **MVP 之前不写意图卡**（会变空想）
- **active 意图 ≤ 12 条**（超出强制合并 / falsify）
- **证伪条件必须可观测**（数据库 / 指标 / 日志能查出数字）
- **user override 是紧急通道，不是默认**（窗口内比例 > 50% 视为范式失败）

## 5. 场景 B：已有源码

### 5.1 起点

代码已经跑在生产，可能伴随：过时的 PRD、不完整的注释、缺测试、有合规约束、团队多人协作。

**比空项目更复杂的地方**：

- 代码里有大量"未文档化的隐性约束"——AI 容易把"过度设计"当"待简化目标"，但实际是历史合规要求
- 已有 PRD 可能严重脱节——AI 把过时假设当真，做出实际不对齐的规划
- 团队已有自己的工作流——引入 AI agent 范式有兼容性问题

### 5.2 反模式

| 反模式                     | 后果                         |
| ----------------------- | -------------------------- |
| "Claude，扫一下代码自动生成规格"    | AI 写出表面准确实际错误的规格（识别不到隐性约束） |
| 一上来开 9 个 Agent 并行重构     | 没有测试网络的项目会被改崩              |
| 把已有 PRD 直接喂 AI          | AI 把过时假设当真                 |
| 跳过 Phase 0 直接挂意图卡       | 老代码无尽 backlog              |
| Phase 2 测试网络没建好就让 AI 大改 | 看似对实际崩，且没有回滚锚点             |

### 5.3 Phase 0 · 考古（1-2 周）

**输入**：已有代码 + 可能过时的 PRD

**输出**：`docs/execution/CODEBASE-MAP.md`（带人类批注的代码反向地图）

**Claude Code 角色**：考古者（**只列事实，不写"为什么"**）

**工作流**：

1. 派 `Explore` agent 扫描，产出地图：路由表 / 领域模型 / API 端点 / 依赖图 / 主要数据流 / 测试覆盖现状
2. **人类逐段标注**：哪些是核心？哪些是技术债？哪些是合规要求不能动？哪些是死代码？
3. 把"为什么这段代码存在"显性化到 batch-0 ADR

**关键风险**：AI 容易把"过度设计"当"待简化目标"。比如一个 5 层 if 嵌套，看起来该重构，实际可能对应两年前的 GDPR 整改。**人类标注是不可省的**。

**完成判定**：

- [ ] CODEBASE-MAP.md 覆盖核心模块
- [ ] 每个核心模块有"为什么存在"批注
- [ ] 已识别的隐性约束 ≥ 5 条入 DECISIONS.md（追溯 ADR）

### 5.4 Phase 1 · 反推意图（1 周）

**输出**：`docs/execution/INTENT-GRAPH.md`（≤ 10 条反推意图，**先不启用证据通道**）

**两类关键产出**：

| 类型          | 含义                 | 行动                          |
| ----------- | ------------------ | --------------------------- |
| **代码有但没意图** | 疑似过度设计 / 死代码 / 技术债 | 写"待证伪意图"——若 6 周无人维护无人调用 → 删 |
| **意图有但代码弱** | 产品弱点 / 已知问题        | 写普通意图卡，进入证据通道               |

**Claude Code 工作流**：

1. 基于 CODEBASE-MAP，AI 起草反推意图清单
2. 用户审：哪些是真意图、哪些是 AI 误读
3. 定稿入 INTENT-GRAPH，但**状态全部 pending**——证据通道等 Phase 2 测试网建好

### 5.5 Phase 2 · 建机器可读契约网络（2-6 周，**最不能省**）

**这是已有项目最大的工作量，但跳过会出大事故**。

**核心目标**：把已有代码的隐性约束显性化为机器可执行。

| 工具类  | 作用                                                    |
| ---- | ----------------------------------------------------- |
| 类型检查 | mypy / pyright / TypeScript strict                    |
| 测试套件 | pytest / vitest（先补核心路径基线）                             |
| 架构守门 | importlinter / ESLint boundaries / dependency-cruiser |
| 安全扫描 | bandit / semgrep                                      |
| 决策日志 | DECISIONS.md（追溯历史决策）                                  |

**Claude Code 工作流**：

1. 派 `general-purpose` agent 增量补测试（**先核心路径**，不追求 100% 覆盖）
2. 加 lint / typing 配置，逐文件修复（用 `Agent` 派批量修复）
3. 跑 `git log --since=...` 让 AI 起草 ADR 草稿，**人类审定后**入 DECISIONS.md
4. **不要并行启 9 个 agent 改代码**——没有测试网的项目会崩。Agent Team 并行度从 2-3 开始

**完成判定**：

- [ ] 核心路径测试覆盖率 ≥ 60%
- [ ] CI 全绿
- [ ] importlinter（或等价工具）能跑
- [ ] DECISIONS.md ≥ 10 条追溯 ADR

### 5.6 Phase 3 · 增量挂意图（永远不结束）

**触发条件**：Phase 2 测试网络建好。

**规则**：

- 所有**新 feature / 改动**强制挂意图卡（INTENT-GRAPH §1.3 字段清单）
- **老代码维持现状**——直到有人触碰时才反向补意图
- 6-12 周后意图图谱会逐步覆盖代码库

**Agent Team 工作流**（已有项目专属）：

- **并行度低**（2-3 agent，互相 review）
- **每次改动派独立诊断 agent 做反向核验**——已有项目里"看起来对实际错"的概率高得多
- **每个 PR 必须报告"动了哪些已有约束"**——避免无声破坏隐性合同

## 6. 关键差异与共性

### 6.1 差异表

| 维度                         | 空项目                | 已有源码               |
| -------------------------- | ------------------ | ------------------ |
| Phase 0 关键产出               | 顶层意图（前瞻 ≤ 5 条）     | codebase 考古地图（回顾）  |
| Phase 1 关键产出               | 最小架构契约（冻结一段）       | 反推意图清单 + 隐性约束显性化   |
| Phase 2 关键产出               | MVP 跑通 + 契约网络（同步建） | 契约网络（先于任何 AI 改造）   |
| Phase 3 启用 INTENT-GRAPH 时机 | MVP 主链路通           | 测试覆盖 ≥ 60%         |
| Agent Team 并行度             | 高（3-5，最高 9）        | 低（2-3 + 互相 review） |
| Claude Code 主角色            | 起草者 + 执行者          | 考古者 + 增量改造者        |
| 主要风险                       | 架构选型错误             | AI 误解隐性约束          |
| 测试套件优先级                    | 与代码同步建             | **先于任何 AI 改造**     |

### 6.2 共性原则（永远成立）

| 原则                        | 理由                        |
| ------------------------- | ------------------------- |
| 项目纪律必须入仓库（不入 memory）      | 仓库可移植，memory 绑机器          |
| Phase 0/1 不能纯意图驱动         | AI 没素材可推                  |
| INTENT-GRAPH 是 Phase 3 工具 | Phase 0 启用会写空想意图          |
| 架构选型永远人类拍板                | 不可逆决策不能交给 AI              |
| 机器可读契约是真相源（非 docs）        | docs 会腐烂，pytest / mypy 不会 |
| Agent Team 并行度看上下文风险      | 风险高 → 并行度低                |
| 用户验收不可省                   | "AI 报告全绿"≠"真的对"           |
| 决策必须留 ADR                 | 6 个月后没人记得为什么              |

## 7. Agent Team 使用指南

### 7.1 什么时候开 Agent Team

**应该开**：

- 跨层实现（同一 Batch 要动 domain + application + infrastructure + interfaces）
- 可并行的独立工作（后端 + 前端 + 测试）
- 多步骤项目（研究 → 规划 → 编码 → 校验）
- ≥ 2 个独立子任务

**不应该开**：

- 单文件小改
- 纯查询 / 探索
- Phase 0 / Phase 1（没有可并行的素材）
- Phase 2 测试网络未建好的已有项目

### 7.2 并行度选择

| 项目类型              | 推荐并行度              | 理由                |
| ----------------- | ------------------ | ----------------- |
| 空项目 Phase 2 主链路   | 3-5                | 有测试网兜底            |
| 空项目 Phase 3 实施意图卡 | 3-5                | 同上                |
| 空项目领域建模等独立子任务     | 5-9                | NovelForge B1 验证过 |
| 已有项目 Phase 2 补测试  | 2-3                | 没有兜底              |
| 已有项目 Phase 3 改老代码 | 1-2 + review agent | 隐性约束风险大           |

### 7.3 多 Agent 并行命名一致性

NovelForge 经验：B1 9 agent 并行建领域，曾经差点同概念多个名字。**必须**：

- 启动前先 grep 现有代码看本概念是否已有命名
- 复用 `__init__.py` 的导出符号；不要起新名
- 跨 agent 共享的术语必须先在 `DECISIONS.md` 出现过；没有就向用户确认
- 合并 PR 前与同 Batch 其他 agent 的命名做一次 diff

## 8. 现实预期：加速幅度

NovelForge 18 Batch / 37 小时墙钟跑通**是特例**：

- 用户提前已有 17 份完整规格（Phase 0/1 在仓库创建前完成）
- 项目复杂度可控（单产品线 SaaS）
- 单一裁决人决策链路短
- 用户技术深度足够，能直接审 ADR

真实企业项目通常没有这些条件——多人协作、合规需求、历史包袱、跨部门决策、不确定的市场反馈都会拖节奏。

**真实加速幅度**（基于 NovelForge + 一般业内观察的估计，**未经独立验证**）：

- 空项目 MVP：30-50% 加速（vs 纯人类团队）
- 已有项目改造：10-30% 加速（隐性约束风险吃掉部分增益）
- 重复模式（CRUD、表单、API 接口）：50-80% 加速
- 创新性工作（架构 / 算法 / 产品判断）：\~0% 加速，AI 是辅助

**任何声称"AI 让你 10x 速度"的工具营销都没把 Phase 0/1/2 算进去**——它们只算了 Phase 3 的代码生成速度。

## 9. 反模式速查表

无论空项目还是已有项目都**不要**：

| 反模式                        | 后果              |
| -------------------------- | --------------- |
| 让 AI 扫一下自动生成完整规格           | 充满想象的虚假 PRD     |
| Phase 0 启用 INTENT-GRAPH    | 空想意图            |
| 跳过 ADR / DECISIONS.md      | 隐性决策必死          |
| 没有契约网络就开 Agent Team 大改     | 看似对实际崩          |
| 把项目纪律写进 memory             | 拷贝丢失            |
| 把 user override 当默认通道      | 退化为带额外文档负担的 PRD |
| 证伪条件用主观词（"用户体验差"）          | 永远证伪不掉          |
| 让单一 Agent 跨多 Batch 上下文连续工作 | 上下文爆炸           |
| 不验收就 commit AI 产出          | "AI 报告全绿"≠"真的对" |
