# AI时代的编程范式革命：从"人类逻辑PRD"到"AI-native自主规划"

## 一、核心诊断：传统PRD的"人类逻辑陷阱"

当你让Claude Code或类似工具写需求文档时，它为什么会输出**人天估算、甘特图、阶段里程碑**？因为这些工具的训练数据来自过去20年的软件工程实践，它们内化了"人类团队需要协调、需要排期、需要资源分配"的隐含假设。

但在AI Agent可以自主编码、测试、部署的场景下，这些元数据变成了**认知噪音**：

| 传统PRD的人类逻辑                   | 在AI-native场景下的冗余性                                    |
| ----------------------------------- | ------------------------------------------------------------ |
| **人天估算**                        | AI不需要休息，执行时间取决于算力而非人力，估算失去意义       |
| **阶段划分（需求→设计→开发→测试）** | AI可以并行执行这些活动，阶段门控反而降低收敛速度             |
| **资源分配与角色定义**              | 多Agent系统自带角色分化，无需预先指定"前端工程师"或"测试工程师" |
| **静态文档作为交接物**              | AI需要持续更新的"活规范"（Living Spec），而非冻结的PRD       |

正如McKinsey在2026年4月的报告中指出的，AI软件开发的最高层级（Level 4）是**"由小型团队引导协调的AI Agent系统端到端交付应用"**，人类的角色是"声明高层意图和边界"，而非进行传统项目管理 。

## 二、AI-native规划范式的四大特征

基于对当前"意图驱动工程"（Intent-Driven Engineering）和Multi-Agent Orchestration的研究，AI-native的规划逻辑应呈现以下特征：

### 1. 意图驱动（Intent-Driven）而非任务分解

传统PRD回答"做什么"和"怎么做"，AI-native规范回答**"期望什么结果"和"受什么约束"**。

> "Intent-Driven Engineering asks: What outcome is desired, and how should the system continuously align to it?" 

**实践形态**：开发者提供"可证伪的意图声明"——例如"我们相信用户会因为X功能而留存，如果观察到Y行为则假设错误"，AI Agent据此自主推导实现路径 。

### 2. 可执行规范（Executable Specifications）而非静态文档

静态PRD在AI写下第一行代码时就开始腐烂。AI-native的规范必须是**可执行的、自同步的**：

- **Living Specs**：规范与代码双向绑定，代码变更自动更新规范，规范变更驱动代码重构 
- **机器可读优先**：使用OpenAPI Schema、TypeScript接口、属性测试（Property-based Tests）作为"真相源"，而非Markdown描述 

> "The specification is the primary artifact, and code is entirely derived from it." 

### 3. 运行时自适应（Runtime Adaptation）而非 upfront planning

人类需要 upfront planning 是因为返工成本高。AI的迭代成本极低，因此规划应该是**运行时涌现的**：

- AI Agent在编码过程中持续发现依赖关系，动态调整执行顺序
- 遇到未预期的技术约束时，自主回溯并重新规划，无需人类重新"排期"
- 采用"Plan → Spawn → Monitor → Verify → Integrate → Retro"的工厂模型，而非瀑布模型 

### 4. 多Agent协作规划（Multi-Agent Orchestration）而非单线执行

单一AI Agent很快会遇到上下文天花板。AI-native开发本质上是**多Agent交响乐**：

| 模式                     | 说明                                                         | 来源 |
| ------------------------ | ------------------------------------------------------------ | ---- |
| **Orchestrator-Workers** | 协调Agent分配任务给专业Agent（后端、UI、安全），并解决冲突   |      |
| **Fan-out/Fan-in**       | 并行派发多个Agent处理不同维度（如安全审计、测试生成、文档同步），再聚合结果 |      |
| **Swarm**                | 扁平化协作，Agent基于能力动态交接任务，适合探索性任务        |      |

Addy Osmani在2026年3月的研究中强调：**"你不再与单个AI结对编程，而是在管理一个Agent团队"**——开发者从"指挥家"变为"管弦乐编排者" 。

## 三、当前工具链的断层分析

2026年的工具生态正处于**范式转换的中途**，存在明显的断层：

| 层级         | 代表工具                                    | 局限                                          | 缺失的AI-native能力                     |
| ------------ | ------------------------------------------- | --------------------------------------------- | --------------------------------------- |
| **IDE层**    | Cursor, Windsurf, Claude Code               | 要求人类提供详细指令，单Agent上下文有限       | 缺乏自主规划层，仍依赖人类写PRD         |
| **规范层**   | GitHub Copilot Workspace, Intent            | 生成静态spec后由Agent执行，spec与代码可能脱节 | Living Specs能力尚在早期                |
| **编排层**   | CCteam-creator, Orrery, LangGraph AgentFlow | 需要人类定义Agent角色和计划结构               | 缺乏从"纯意图"到"多Agent计划"的自动推导 |
| **基础设施** | MCP (Model Context Protocol)                | 提供工具集成标准，但不解决规划逻辑            | 缺少AI-native的"需求理解协议"           |

**关键矛盾**：现有工具让你"用自然语言描述需求→AI生成代码"，但跳过了最关键的环节——**AI对需求的自主解构与规划**。你仍然需要把人类的项目管理思维翻译成AI能理解的指令。

## 四、未来架构：从"写文档给AI看"到"AI自主规划"

基于研究，我构建了一个**AI-native开发架构**的框架：

### 架构核心：三层分离

```
┌─────────────────────────────────────┐
│  Layer 3: Intent Layer (意图层)      │
│  - 人类输入：业务目标、约束、成功标准    │
│  - 输出：可执行意图图谱               │
├─────────────────────────────────────┤
│  Layer 2: Orchestration Layer (编排层)│
│  - AI自主分解目标为子任务              │
│  - 动态分配专业Agent                 │
│  - 运行时冲突解决与重规划              │
├─────────────────────────────────────┤
│  Layer 1: Execution Layer (执行层)   │
│  - 代码生成、测试、部署、监控          │
│  - 持续向编排层反馈状态               │
└─────────────────────────────────────┘
```

### 关键机制

1. **意图图谱（Intent Graph）**：用图结构表示业务目标、用户行为假设、技术约束之间的关系，而非线性文档。AI Agent可以在此图谱上执行**假设验证**（Hypothesis Validation）。

2. **自举式规划（Bootstrapped Planning）**：AI Agent接收高层目标后，不立即执行，而是先进入"发现模式"（Discovery Mode）：
   - 扫描现有代码库理解上下文
   - 识别技术债务和架构约束
   - 生成`MULTI_AGENT_PLAN.md`作为共享真相源 
   - 人类审查计划后，Agent自主执行

3. **持续对齐循环（Continuous Alignment Loop）**：
   - 代码变更 → 自动更新Living Spec
   - 运行时指标 → 反馈到意图层验证假设
   - 假设失效 → 触发重规划，无需人类重写PRD

## 五、实践路径：从"人类逻辑"到"AI逻辑"的转型

### 立即可以采用的策略

**1. 替换PRD为"意图声明+可执行契约"**
- 停止写包含人天估算的PRD
- 使用TypeScript接口/OpenAPI定义行为契约
- 编写"可证伪的成功标准"而非"功能列表" 

**2. 引入多Agent编排**
- 使用Orrery或CCteam-creator将单一AI会话升级为Agent团队 
- 建立`AGENTS.md`定义团队规则

**3. 建立"工厂模型"**
- Plan（意图）→ Spawn（启动Agent团队）→ Monitor（异步检查）→ Verify（质量门控）→ Integrate（合并）→ Retro（更新规则）
- 设置WIP限制：同时运行的Agent不超过3-5个，审查是瓶颈而非生成 

**4. 采用混合范式**
- **探索期（<3个月/MVP）**：使用Vibe Coding快速验证意图 
- **生产期（>3个月/维护）**：切换到Spec-Driven，建立Living Specs作为持久真相源 

### 2026-2027的关键指标转变

| 过时指标         | AI-native替代指标                            |
| ---------------- | -------------------------------------------- |
| 故事点/人天/速度 | 假设验证率、意图到验证学习的时间             |
| 代码行数         | AI杠杆比率（单位人类创意努力的产出）         |
| 缺陷密度         | 规范质量分数（实现与意图的匹配度）           |
| 发布频率         | 意图收敛速度（从目标声明到可工作原型的周期） |

## 六、结论：范式转移的本质

你观察到的现象触及了一个深层转变：**软件工程正在从"人类协调人类"转向"人类协调AI"**。

在这个新范式中：
- **稀缺资源不再是"实现能力"**（AI可以无限生成代码），而是**"意图清晰度"**和**"系统判断力"**
- **开发者角色从"写代码的人"变为"设计工厂的人"**——设计生产AI软件的工厂 
- **文档的本质从"给人看的交接物"变为"给AI执行的契约"**

正如Gartner预测的，到2026年底60%的新代码将由AI生成，到2027年这一比例将超过80% 。在这个趋势下，坚持在AI规划中嵌入"人天估算"和"阶段门控"，就像给自动驾驶汽车配备马车夫的鞭子——不仅多余，而且会阻碍AI发挥其自主规划的潜能。

**未来的需求文档不是PRD，而是Intent；不是静态的，而是Living；不是给人类阅读的，而是给AI执行的。**
